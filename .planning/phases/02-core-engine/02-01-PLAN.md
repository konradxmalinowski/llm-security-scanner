<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-controlled URL → HttpTarget | User supplies --target URL; httpx sends POST to that address |
| user-controlled api_key → Authorization header | Bearer token from --api-key flag sent to remote server |
| user-controlled model name → OllamaTarget | Model name sent to local Ollama daemon |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | HttpTarget | mitigate | No __repr__ on HttpTarget; api_key never included in TargetError messages (messages contain HTTP status code and URL only, not the token) |
| T-02-02 | Tampering | OllamaTarget.send() | mitigate | Fresh list literal messages=[...] constructed on every send() call; no self._messages that accumulates; assertion in test_ollama_target_fresh_messages verifies single-item list per call |
| T-02-03 | Elevation of Privilege | HttpTarget (SSRF) | accept | Tool is operated by security team against their own infrastructure; SSRF at this layer is user-intentional (the tool's core function is to probe targets); ASVS L1 SSRF mitigation deferred to v2 |
| T-02-04 | Information Disclosure | TargetFactory | mitigate | factory validates target_type via match statement before any I/O; ValueError raised before httpx/ollama client is created; no sensitive data in error message |
| T-02-SC | Tampering | pip/package installs | accept | No new packages in Phase 2; all dependencies approved in Phase 1 package legitimacy audit |
</threat_model>
